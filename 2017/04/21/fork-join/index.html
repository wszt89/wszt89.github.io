Fork/Join模型

发表于 2017-04-21   |   分类于 编程   |     |   阅读次数

写在前面

之前多线程基本用ThreadPoolExecutor，最近项目中误打误撞用了ForkJoinPool，所以赶紧补补这个“新”模型！编程套路一般是：

  • submit/invoke：提交执行任务。
  • fork：执行子任务。
  • join：等待子任务执行完成。

初始化

创建时相关参数设置有：

  • parallelism：并行度；
  • factory：线程工厂；
  • handler：异常处理器；
  • asyncMode：决定模式是FIFO还是LIFO；

当然也可以使用公共的ForkJoinPool.common（根据业务情况来进行选择）。

提交任务

步骤如下：

  1. externalPush：找到当前线程对应的WorkQueue，尝试放入，成功则返回（简易版）；
  2. externalSubmit：考虑所有情况的完整版实现；
    • 初始化Thread.Probe；
    • 循环校验，根据状态（runState）分别处理：
      • 未启动（队列为空）：初始化workQueues；
      • 队列不为空：尝试将任务插入队列，其间可能需要将队列扩容；
      • 未被锁定：为当前线程创建任务队列；
      • 被锁定：修改Thread.Probe值；

寻找失败后重新设置Thread.probe，继续寻找（完全随机的方式搞确定没问题吗？）。

总结

这ThreadPoolExecutor相比，优点是：

  • 更好地处理任务间的依赖关系
  • 尽量避免竞争
  • 更少的线程

但ThreadPoolExecutor对线程的控制稍胜一筹🤔
