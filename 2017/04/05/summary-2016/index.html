总结：二零一六

发表于 2017-04-05   |   分类于 总结   |     |   阅读次数

在菜鸟的第三个年头，有很多的不如意和更不如意，不管结果怎么样，都值得花一段时间认真地思考和总结。

项目总结

之前一直在做数据，从16年5月开始搞业务，不得不说菜鸟的业务还是重，而仓库里面的业务更重。

退供

这是第一个深入接触的业务，流程比较简单：

  • 接单
  • 下架
  • 质检
  • 出库

但是等系统的实现捋了一遍之后，简直丧心病狂，比较玩好的是之前：

系统事务不生效，有问题的数据通过订正数据来解决。

于是，接手的一周迅速从开发工程师“成长”为数据订正工程师，尤其擅长大单据的问题排查。

集货二期

很有意思的一个项目:

集货仓作为一个中心仓，为前置仓供货。

通过不同功能的仓、把原来的点连成图，在供应链层面能玩的东西就很多了。项目过程并不顺利：

  • 业务方的复盘和调整；
  • 中途上游暂停，只有下游的我们在坚持，每周的周报看起来略显尴尬；
  • 系统问题多，修复和订正数据占用了大量的开发时间；
  • 每个人并行项目比较多；

很重的业务，并且同时跟这么多人打交道：

每个人都需要为自己的绩效和利益考虑，况且还有这么多的山头和小团队。

在这样的团队生存，还是需要找一条靠谱的大腿的（一般情况下要找那条最粗的）。

双十一库存兜底及优化

被忽悠着擦了一次屁股，唯一的好处是了解了库存性能的难点，同时熟悉了库存的业务。

库存优化

休完婚假无事一身轻的状态归队，思考后面搞什么，本希望以库存优化作为突破口：

  • 爆品优化
  • 滞销品优化
  • 碎片整理

其中负责了整体的前期推进和前端开发，本希望半个月到一个月的时间可以落地，但是拖拖拉拉地搞了好几个月。另外：

之前想的长期方案是和算法团队深度合作，但是没想到刷存在感的版本变成了最终版。

结局是：然后就没有然后了。

库存分配

如果库存只在性能优化的路上一直走，容易死掉：

  • 越来越薄、越来越简单
  • 背靠阿里强大的中间件，异步削峰以后的流量不是那么夸张

比较难并且需要长期思考的事情是：

  • 作业效率的提高
  • 不同作业之间的协调

库存分配是单据数据和库存数据的深加工，在目前的系统中应该算是超智能化迈出的重要一步，认真搞搞还是蛮有前途的。落地上：

  • 2C部分路径优化部分调用算法VRP
  • 2B部分完全自己实现

整体的感觉是：

  • 流程太长、业务太复杂，我还是比较想做VRP这种事情(ノへ￣、)
  • 以后算法要白盒
  • 项目有点混乱

做的过程中，一直也在思考：

库存分配后面要怎么做、达到什么样的效果？

对比一下MySQL和WMS，MySQL负责数据的存取、WMS负责货的进出，SQL语句有优化引擎，WMS的作业应该也需要这么一个智能中心。

技术成长

工作中用到的基本都是Java基础：

试了一把jdk8的Lambda表达式感觉还不错，尴尬的是出来这么久了，现在才开始用。有时候也在反思自己是不是有点老了，对新东西没那么大兴趣了。

总体来说，入了业务的坑之后，技术能力稳定下滑中。机器学习参加了个比赛，结果被血虐，收获是：

  • 学会了用WEKA；
  • 明白了机器学习并不只是一个模型，而是清理、加工、训练、验证等一整套流程；

本着知其然也要知其所以然的目标，学习了一些“传统”的机器学习算法，很多算法的年龄比我的年龄都大。过程中深刻体会到数学的重要性，尤其是：

  • 线性代数
  • 概率论
  • 运筹学

多打打基础，深入思考工具和算法背后的原理，有时候“慢慢来，反而快”。

总结与规划

生活还是细水长流、平淡无奇地美好：

很久没时间打DOTA，偶尔慢跑，羽毛球也渐渐生疏了。

再过不久就要升级了，除了经济上的、最大的压力在于如何去做一个榜样或值得尊敬的人，最起码应该是：

  • 有理想与追求
  • 有底线和原则

然后，像“疯狗”一样去追求自己的理想，像“疯狗”一样去守护自己的底线。
