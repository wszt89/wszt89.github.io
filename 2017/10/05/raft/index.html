RAFT协议

发表于 2017-10-05   |   分类于 算法   |     |   阅读次数

问题是什么？

分布式的网络中的通信是不可靠的，那通过“不靠谱”的网络是否能达成一致的结果？这就是“拜占庭问题”：

拜占庭幅员辽阔，军队间距离很远，只能靠信差传递消息。在战争时期，所有将军必须达成一致共识，才去攻打敌人。

然而，军队中存在间谍和叛徒，此时，忠诚的将军们如何才能不受影响并达成一致。

拜占庭是对现实世界的模型化，拜占庭失效是指：一方向另一方发送消息，另一方没有收到或收到错误的消息。

PAXOS

Lamport也讲了一个故事，有三种角色：

  • PROPOSER：提出提案，内容包括编号$N$和$VALUE$；
  • ACCEPTOR：接受提案；
  • LEARNER：学习被选出的提案（毫无参与感）；

算法分成准备和批准两阶段：

  • 准备阶段
    • PROPOSER选择一个提案编号$N$，发送给ACCEPTOR中的一个多数派；
    • ACCEPTOR收到请求后，如果$N>M$，则将上次接受的$VALUE$回复给PROPOSER，并承诺不再回复小于$N$的提案；
  • 批准阶段
    • PROPOSER收到多数派的回复后，向对应的ACCEPTOR再次发送请求（编号依旧是$N$，如果准备阶段有返回$VALUE$则使用，否则创建一个）；
    • ACCEPTOR收到请求后，如果编号$N$大于本地收到过的编号$M$，更新$M$和$VALUE$，并返回$M$；

算法保证了：

  • 如果一个编号为$N$的提案的值为$VALUE$，那么存在一个多数派，要么他们中任何人没有接受编号小于$N$的任何提案，要么他们已经接受的所有编号小于$N$的提案中编号最大的值为$VALUE$；
  • 当且仅当ACCEPTOR没有回应过大于$N$的准备阶段请求时，ACCEPTOR才会接受编号为$N$的提案；

RAFT

算法中包含三种角色：

  • LEADER：领导，最多一个；
  • FOLLOWER：被领导通知干活的；
  • CANDIDATE：从FOLLOWER到LEADER的中间状态；

变化如下：

选举逻辑：

  • LEADER定时给FOLLOWER发送心跳消息；
  • FOLLOWER收到LDEADER的心跳消息后，重置选举定时器，更新自己的TermId；
  • FOLLOWER的选举定时器超时后，转变为CANDIDATE并发起投票；
  • CANDIDATE收到过半投票后变为LEADER；
  • CANDIDATE没有收到足够的票数，重新设置选举定时器避免冲突，自己的TermId加一；
  • FOLLOWER收到选举消息后，如果没有投票就投票（根据TermId来判断是否投过），并重置自己的选举定时器；

写入逻辑：

  • CLIENT发送请求给LEADER（不能给其他的角色发送请求）；
  • LEADER将请求记录到日志，并发送给其他服务器；
  • LEADER收到过半服务器的保存响应，请求提交并通知CLIENT成功；
  • LEADER刚被选举出来以后，以LEADER的日志为准（LEADER需要有所有已提交的日志，投票阶段需要比较日志中的TermId），并同步给FOLLOWER；

集群变化（分两阶段防止出现多个LEADER）：

  • LEADER收到命令，从$C_{old}$切换到$C_{new}$；
  • LEADER本地生成一条日志，内容为$C_{old}\cup C_{new}$，并推送给FOLLOWER；
  • FOLLOWER收到请求后，将$C_{old}\cup C_{new}$作为配置；
  • LEADER在收到大多数的回复后，提交该记录；
  • 接着，LEADER生成新的一条记录$C_{new}$，并推送给FOLLOWER；
  • FOLLOWER收到请求后，将$C_{new}$作为配置；
  • LEADER收到大多数的回复后，返回CLIENT成功；
