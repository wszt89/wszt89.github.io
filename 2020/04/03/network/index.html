TCP

协议头

  • 源端口&目标端口：分别用来找到发端和收端的应用进程；
  • 序号：发送方将数据分成多段发送给接收方，在每次发送时需要告诉接收方这次发送的是哪一段，在达到最大值后会继续从0开始；选择随机号防止被攻击；
  • 确认号：接收方告诉发送方需要发送的下一段数据的序号，也就告诉对方前面的数据我已经收到了；
  • 数据偏移：标记可选项大小；
  • 保留：略；
  • 状态控制码
    • URG：配合紧急指针使用，通知系统有个数据是紧急的；
    • ACK：配合确认号使用；
    • PSH：控制读取或者发送数据的时机，其他的比如缓存区满了也会读写；
    • RST：出现严重错误需要释放连接并重新建立；
    • SYN：标记这是一个连接请求或者连接被接受的报文；
    • FIN：标记发送端数据传送已完毕，并要求释放连接；
  • 滑动窗口：告诉对方自己能接收数据的大小，避免发送的数据过大无法被接收，滑动窗口的大小是动态的；
  • 校验和：用来发现数据是否损毁；
  • 紧急指针：标记紧急字节的位置；
  • MSS：和IP层的分片大小有关；

三次握手

  • client->server：SYN、seq=J；
  • server->client：SYN、ACK、seq=K、ack=J+1；
  • client->server：SYN、ACK、ack=K+1；

四次挥手

  • client->server：FIN、seq=M；
  • server->client：ACK、ack=M+1，到这里是半关闭状态；
  • server->client：FIN、seq=N；
  • client->server：ACK、ack=N+1；

慢启动

目的：

当TCP开始在网络中开始传输（或者重发）数据时，会慢慢地对网络容量进行试探，避免发送过量的数据导致阻塞。

实现：

发送方TCP增加拥塞窗口（CWND），在收到用ACK消息后增加窗口大小，当发生超时或者重复确认后减小窗口大小。

计时器

  • 重传计时器：发送消息开始一定时间，没有收到ACK则重传；
  • 持久计时器：发送和接收窗口都为0时会死锁，所以发送方会定时发送探测报文；
  • 保活计时器：客户端发送数据后保持沉默，服务端需要知道客户端的死活，于是发送探测报文，如果探测报文连续不回复则会强行中断；
  • 时间等待计时器：四次挥手关闭连接是用来等待；

Nagle

问题：

如果发送的数据很小，数据中有用的信息占比很小，大部分是协议数据（TCP头部占20字节，IP头部再占20字节）。在广域网上会增加拥堵。

思路：

任意时刻最多只有一个未被确认的小段（小于MSS尺寸的数据）。

判断具备发送的条件有：

  • 长度达到MSS；
  • 包含FIN；
  • 设置了TCP_NODELAY选项；
  • 未设置TCP_CORK选项，且数据包都已经被确认；
  • 超时，一般是200毫秒；

可以发现Nagle算法带来的副作用也比较明显：增加了延迟，所以框架基本上都会禁用掉。

延迟确认

与Nagle不同，Delay ACK试图从接收方来优化：

接到报文后不会立即ACK，而是等一段时间看有没有一起发送的数据，目标也是减少数据包。

和Nagle类似，开启后会增加延迟。

如果同时开启可能的效果是：发送方在等ACK，而接收方在等新的数据，于是可能耗时就明显加长了。

保活机制

客户端可能因为各种原因断开，如果服务端不知道会一直维护着连接，时间长了以后大量的这种连接会占用很多资源：

在空闲、没有数据传输时，定时发送确认包来探测。

通过套接字设置SO_KEEPALIVE生效，相关参数：

  • tcp_keepalive_time：最后一次数据传输到发送探测包的间隔，也就是空闲时间；
  • tcp_keepalive_probes：探测次数；
  • tcp_keepalive_intvl：连续探测的间隔时间；

Tips：

相比较，HTTP的Keep-Alive的作用是和服务器交互完后不会直接关闭连接，避免每个请求都建连接。

大量CLOSE_WAIT
  • 原因：
    • 客户端关闭SOCKET发送FIN，服务端接收FIN发送ACK处于CLOSE_WAIT状态；
    • 理论上会马上发送FIN变为LAST_ACK状态；
    • 如果服务端SOCKET对应的线程夯住了，那么由于TCP是双工的，可能服务端还有东西给客户端，这时候系统不能自动关闭，要等线层醒了再关；
  • 解法：
    • 保活机制；
    • 在对方关闭连接时read返回0/-1，使用epoll也可以拿到，这时候可以关闭连接；但是如果自己的线程夯住就没办法了；
大量TIME_WAIT 最大连接数
  • 客户端：需要绑定自己一个空闲的端口，每个连接都不一样，所以限制来源于协议里的端口号长度$2^{16}$。
  • 服务端：绑定端口后，在这个端口上可以不停地获取连接；

LVS

章文嵩博士主导的开源项目，用来做负载均衡，后被集成到Linux内核中。服务器角色有：

  • Director：调度器、网关；
  • Real Server：服务器、真正提供服务的机器；

具体实现在IP、TCP层搞。

模式

LVS-NAT

流程：

优点：

  • 支持端口映射；
  • RS可以使用任意的操作系统；
  • 节省公网IP，后端的主机较安全：RIP和DIP使用同一网段私有地址，RS的网关要指向DIP；

缺点：

  • 请求和响应都需要经过Director，这样单点的压力会比较大；

LVS-TUN

流程：

优点：

  • RIP、VIP、DIP都是公网地址，RS网关不指向DIP；
  • Director只处理进站请求，不处理响应；

缺点：

  • 不支持端口映射；
  • 需要支持隧道功能，同时隧道功能会消耗性能；

LVS-DR

流程：

优点：

  • Director仅处理请求，不处理响应；
  • RS可以使用大多数OS；

缺点：

  • 不支持端口映射；
  • 不支持跨局域网；

编程

了解一些协议后，可以开始继续看编程的使用需要注意些什么，到应用层了。在UNIX中定义了五种I/O模型：

  • 阻塞I/O
  • 非阻塞I/O
  • I/O复用
  • 信号驱动I/O
  • 异步I/O

tips：

同步和异步的区别在于：调用者是否主动等待结果；阻塞和非阻塞的区别是：程序在等待结果时的状态。

Linux

方法：

  • socket：创建网络通信使用的套接字；
  • bind：绑定；
  • listen：套接字由主动改为被动，监听连接；
  • accept：取出一个已经建立的链接，如果没有连接可用则阻塞；
  • connect：与服务端建立连接；
  • read&write&recv&send：读写；

在建立连接后，可以在主线程里面操作进行读写，但是这样会导致其他的连接无法响应。通常的做法是：

用线程池来处理连接。

在存活的连接数大于线程数时，同样会阻塞主线程。而且从服务器来看单个连接很多时间是不活跃的，这样就有了更高级的多路复用玩法：

  • select
  • poll
  • epoll：事件通知方式，在fd就绪时，系统注册的回调函数就会被调用；不需要把数据频繁在用户态和内核态间互相拷贝；

这样的话线程数只与活跃的连接数有关系，大大减少了线程的数量。

命令

netstat

用来显示各种网络相关的信息，比如连接、路由、状态等：

  • netstat -a：所有连接；
  • netstat -at：TCP连接；
  • netstat -l：处于监听状态的连接；
  • netstat -p：显示进程ID和名称；
  • netstat -s：统计；

dstat

动态监控各种指标，还是彩色的哦：

  • dstat --socket 5 5：tcp和udp端口状态；
  • dstat -n 5 5：网络流量；

ping

用ping 127.0.0.1来检查到目标地址的网络是否是通的。
